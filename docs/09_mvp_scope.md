# 09. MVPスコープ

## MVPの目的

ClassFlow Mini のMVPは、教室管理のすべてを作ることではありません。

MVPの目的は、以下の流れを動かすことです。

```text
生徒登録
↓
クラス登録
↓
レッスン作成
↓
欠席登録
↓
振替チケット発行
↓
振替予約
↓
出欠登録
↓
未消化振替確認
```

この流れが動けば、ClassFlow Miniの核は完成です。

---

## MVPで作る機能

| 優先度 | 機能 | 説明 |
|---|---|---|
| 1 | ログイン | 管理者がログインする |
| 2 | 生徒管理 | 生徒・保護者情報を登録する |
| 3 | クラス管理 | 曜日・時間・定員を登録する |
| 4 | クラス所属管理 | 生徒をクラスに所属させる |
| 5 | レッスン管理 | 実際の日付のレッスンを作成する |
| 6 | 欠席登録 | 欠席情報を登録する |
| 7 | 振替チケット発行 | 欠席時に振替権利を発行する |
| 8 | 振替予約 | 未使用チケットを別日に割り当てる |
| 9 | 出欠登録 | レッスン当日の出欠を登録する |
| 10 | 未消化振替一覧 | 残っている振替を確認する |

---

## MVPで作らない機能

以下は後回しにします。

| 機能 | 後回し理由 |
|---|---|
| 保護者ログイン | 実装範囲が広がる |
| 保護者用欠席フォーム | MVP後に追加でよい |
| LINE連携 | 設定・審査・運用が重い |
| メール送信 | 最初は不要 |
| 決済 | 個人開発の信用ハードルが上がる |
| 月謝管理 | 会計要素が入って重い |
| 成績管理 | 学習塾寄りになりすぎる |
| 講師給与管理 | 別システムになる |
| 複数校舎管理 | MVPでは不要 |
| 複雑な振替ルール | 初期実装が難しくなる |

---

## MVPの割り切りルール

MVPでは以下のルールに固定します。

### 欠席と振替

```text
1欠席 = 1振替チケット
```

### 振替期限

```text
欠席日から30日後
```

### 振替可能クラス

MVPでは同じ教室内の空きがあるレッスンを候補にします。

レベル制限や曜日制限は後回しにします。

### 満席制御

振替先レッスンの参加予定人数が定員に達している場合、予約できないようにします。

### 保護者連絡

MVPではシステムから送信しません。

先生が画面を見て、LINEや電話で連絡します。

---

## MVPの完成条件

MVP完成条件は以下です。

1. 管理者がログインできる
2. 生徒を登録できる
3. クラスを登録できる
4. 生徒をクラスに所属させられる
5. レッスンを登録できる
6. 欠席を登録できる
7. 欠席登録時に振替チケットが発行される
8. 未使用チケットを別レッスンに振替予約できる
9. レッスン当日に出欠登録できる
10. 振替参加者が出席したらチケットが消化される
11. 未消化振替一覧で残チケットを確認できる

---

## MVP後の拡張候補

MVPが完成したら、以下の順で追加するとよいです。

| 優先度 | 拡張機能 | 内容 |
|---|---|---|
| 1 | 保護者用欠席フォーム | ログイン不要URLから欠席申請 |
| 2 | 保護者用振替希望フォーム | 空き枠を見て申請 |
| 3 | メール通知 | 欠席受付・振替確定通知 |
| 4 | CSVインポート | 生徒一括登録 |
| 5 | 月次レポート | 欠席数・未消化数の集計 |
| 6 | LINE連携 | 欠席受付や通知 |
| 7 | 月謝管理 | 上位プラン化 |
